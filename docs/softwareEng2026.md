# 第一回

#### ソフトウェア工学の難しいところ
- ソフトウェア開発に携わった経験がないとイメージがわかない。
- 知見を理解しても実践することが難しい。

- ハードウェア・ソフトウェアはセキュリティパッチやバグなどの改良が続く。


# 第二回

・ソフトウェアは情報を扱うプロダクトそのもので、プロダクトを提供する手段のことである。

#### ソフトウェアの定義は
1. 実行されることによって必要な特性、機能、性能を提供する命令語群（コンピュータプログラム）
2. プログラムが適切に情報を扱うことを可能とするデータ構造
3. プログラムの操作や使用法を記述した情報
であり、ハードウェアと異なり劣化しないが悪化する。

#### ソフトウェアの特徴は
- 新しい環境や技術をニーズを満たすよう適応しなければならない
- 新しいビジネス要求を実現するために強化されなければならない
- より近代的なシステムやデータベースと相互運用するため拡張されなければならない
- 進化し続ける環境で実行可能となるよう、再構築されなければならない
というものである。

ソフトウェア工学は「品質」、「コスト」、「納期」の最適なバランスを実現するための手法・方法論のことで
大規模化と複雑さによってソフトウェアなしでは社会がまわらなくなっていたり、ソフトウェアの需要に対して
IT人材が追い付いていなかったり、不確実性によってソフトウェア工学は必要である。


# 第三回

#### ソフトウェアの誕生は
- ニーズの発生
- 要求を具現化するための企画・計画を行う
- 要件定義書をまとめる
という流れで起きる。

要件定義にはシステムの目的・概要・機能・構成や拡張性、開発スケジュールなどが含まれる。実現する機能・実現しない機能を明確にする必要がある。

バグのないソフトウェアはないので、あらかじめテストしてバグを早期に発見・対処することが理想である。


# 第四回

プロジェクトは有期性と独自性を持ち、必ず終わりがあるルーティンワークとは異なる、独自の目的達成のための仕事である。

フォアキャスティングは今の立ち位置から順に目標に進んでいくことであり、目標に到達できない可能性がある。
バックキャスティングは定めた目標に向かって何が必要かを計画し達成に向かうことであり、目標を明確に設定する必要がある。

ソフトウェアにおいて
- 品質優先するのは人命に影響があるシステム
- 費用優先するのは公共関連システム
- 納期優先するのはイベント行事関連システム

バグ発生率とは１ステップ数（改行を除いたコード行数）あたりの発生バグ数のことでバグ発生率はソフトウェア品質評価の1指標となる。

ソフトウェアの評価は
1. コードの物量(ステップ数)
2. コードの物量(オブジェクト容量)
3. ファンクションポイント法(システム規模)
4. 使い勝手(視認性、操作性、互換性など)
によって決まる

### 開発プロセス
- ウォーターフォール型開発プロセス:進捗開発が容易、成果物が明確
- スパイラルモデル:プログラム開発を小さなフェーズに分割する。プロトタイプ作成に想定外の作業量が発生する可能性がある。
- 反復型開発プロセス:ソフトウェアを機能分割する。部分的に完成させていくので顧客の要求を取り入れやすいが作業や業務管理が増え全体像が見えづらい
- アジャイルプロセス:計画を硬直的に守るよりも変化に対応させる


# 第五,六回

WBS（Work Breakdown Structure）とはプロジェクト目標を達成し、必要な要素成果物を生成するために、プロジェクトチームが実行する作業を、要素成果物を主体に階層的に要素分解したもののことで、スコープが明確になり、全体管理と作業計画が明確化されるなどといったメリットがある。

# 第七回

コードは書くことよりも読まれることが多い。そのためソフトウェア開発において読みやすいコードを書くことは必須である。例えばpythonにはPEP8と呼ばれるルールがある。PEP8では例えば
- コードの1行の長さを79字以内とし、コメントは72字以内にする。
- 1レベルインデントするごとに、スペースを4つ使う
- 演算子の前後にスペースを1つずつ入れ、無駄なスペースは入れない
などといったルールがある。また、このルールにのっとったチェックツールもある。(flake8)

# 第八回

バージョン管理とはファイルを誰が、いつ、どのように変更したか管理すること。これらの情報が管理されているならば共同開発ができる。
集中管理型とはリモートリポジトリのみがバージョンを管理する方式で、同時編集するとコンフリクトが発生しやすく、同期に時間がかかる。
分散管理型とはローカルでもバージョンを管理する方式で、リモートリポジトリへのアクセス頻度が低く、障害に頑健である。
gitとは分散管理型のバージョン管理システムでもともとはオープンソースソフトウェア開発のためのソフトウェアであった。特徴として変更履歴が残り、変更した箇所に戻れ、他人と共同編集できるという点がある。
